# Case Study Integration – Parts 3 and 4

This document incorporates **Part 3** and **Part 4** of the Bristol Regional Food Network case study into the current Django marketplace project.

> Scope note: Parts 1 and 2 are assumed to already be completed. This document adds the missing project coverage for Parts 3 and 4 at **design/documentation level only**. No new code, automation, or production functionality has been introduced here.

---

## Part 3 – AI Engineer Model Management

### Requirement Summary
AI engineers should be able to:
- train a new machine learning model outside the system,
- upload the updated model into the service,
- access a database of end-user interactions in order to refine and retrain the model.

### How this is incorporated into the current project
The existing marketplace can support this requirement conceptually through an **AI model lifecycle layer** that sits alongside the web application.

#### Proposed project inclusion
1. **External model training workflow**
   - Model training takes place outside the Django web app.
   - Engineers use exported marketplace data to retrain recommendation or quality-assessment models.
   - This keeps training separate from the live customer-facing platform.

2. **Model upload area (planned feature)**
   - The project now recognises a dedicated administrative area where approved model files could be uploaded and versioned.
   - This would allow the active model to be replaced without rebuilding the whole platform.

3. **User interaction dataset access**
   - Customer browsing, searching, cart activity, purchases, and repeated orders are treated as the core behavioural data source.
   - These records would be made available to AI engineers in a secure and privacy-aware format for retraining purposes.

4. **Governance and version control**
   - Each uploaded model should have a version number, upload date, uploader identity, and deployment status.
   - Older models should be retained for rollback and audit purposes.

### Evidence of fit with the current system
This marketplace already contains the core business entities needed for this requirement, including:
- customers,
- producers,
- products,
- browsing/search behaviour,
- cart interactions,
- account-based access.

These provide the right foundation for future recommendation-model improvement and model replacement workflows.

## Part 4 – Main Administrator Visibility and Explainable AI

### Requirement Summary
The main administrator must have:
- full visibility of platform activity,
- the ability to monitor usage and activity levels,
- an Explainable AI (XAI) feature showing how predictions or recommendations were derived.

### How this is incorporated into the current project
This requirement is now represented in the project as an **administrator oversight and transparency layer**.

#### Proposed project inclusion
1. **Administrative monitoring dashboard**
   - A central admin view would summarise marketplace activity.
   - Example indicators include total users, active producers, product counts, stock warnings, popular items, and customer activity trends.

2. **Operational oversight**
   - The administrator role is responsible for reviewing platform health, monitoring adoption, and spotting unusual behaviour.
   - This supports better decision-making for marketplace growth and inventory planning.

3. **Explainable AI panel (planned feature)**
   - Any recommendation or prediction generated by the AI component should be accompanied by a short explanation.
   - Example explanations could include:
     - frequent past purchases,
     - similar customer behaviour,
     - seasonal demand patterns,
     - product availability,
     - freshness or quality-score rules.

4. **Audit and trust support**
   - Admin users should be able to review which model version produced a recommendation and what inputs influenced it.
   - This improves transparency, trust, and accountability.

## Combined Project Positioning

With Parts 3 and 4 incorporated, the project now covers:
- customer-facing marketplace behaviour,
- producer-side product and inventory management,
- AI model maintenance by engineering staff,
- administrator-level monitoring and AI transparency.

This makes the platform more realistic as an enterprise-style digital marketplace with an AI governance structure rather than only a basic buying and selling site.
